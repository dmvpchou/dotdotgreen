# 點點綠市集完整研究與營運科技藍圖

> 研究日期：2026-07-15（Asia/Taipei）
> 研究範圍：公開網站、Facebook 粉專、臺北捷運／臺北市政府、農業部農糧署、招募頁、媒體文章與政府開放資料。
> 文件目的：建立可供市集經營、北投站營運、新北擴站、異業合作與軟體規劃使用的共同決策基礎。
> 重要限制：本研究只使用公開資料，沒有內部合約、完整班表、攤租、銷售、成本、人力、客訴與事故紀錄；涉及營收、人力與新站選址之結論，需用內部資料與現勘再驗證。

## 1. 執行摘要

### 1.1 研究結論

1. **點點綠已從雙連單點市集變成多站點微型通路。** 2024 年 6 月從雙連站假日市集試營運；2024 年後發展週間 Green to Go；2025 年 7–8 月擴至士林與石牌；Facebook 於 2026 年 6 月 28 日公告北投站 7 月 1 日初登場，7 月 13–17 日週公告已把雙連、士林、石牌、北投四站並列。
2. **模式的差異化在「通勤順手買」而不是攤數規模。** 公開專文描述週間每站約 3–5 攤、每日 5 小時，單攤日營業額曾落在 NT$5,000–22,000；這是媒體／主辦方觀察值，不是經會計查核的保證。市場補上了早市、超市與大型週末市集之間的日常採買空缺。
3. **Facebook 是最新消息中心，但不適合當營運資料庫。** 它能觸及追蹤者、發布週表與臨時休市，卻不提供穩定的結構化班表；官網、招募頁與粉專也已出現北投狀態、週末出口、申請表單與聯絡方式不同步的情況。
4. **北投現在應視為「已上線、待驗證的營運試點」，不是尚未開始的擴站計畫。** 下一步不是再做上線宣傳，而是建立 6–12 週的站點基線，判斷供給、銷售、出席、現場負荷與回購是否能穩定。
5. **跨到新北前，要先把四站營運標準化。** 新北會增加不同場站業主、政府窗口、物流距離、站務規範與客群差異；如果班表、合規、現場事件與數據仍靠訊息和貼文管理，擴點會等比例放大錯誤與溝通成本。
6. **資訊科技最先要解決的是後台營運，不是電商。** 第一優先是攤友主檔、證明文件、日期型班表、場次狀態、現場簽到、異動通知、來源與版本；網站與 Facebook 都應從同一份已核准資料發布。預購、會員、金流與 AI 應在基本資料可靠後再上。
7. **市集具備其他業務的雛形。** 近期公開活動已出現政府減塑合作、新北青農主題、公益夥伴、免費場地招募、試吃與親子體驗，表示可發展 ESG 專案、地方物產週、企業團購／站點取貨、食農教育、社福共融與影響力報告。

### 1.2 建議的近期決策

- 立即把官網現況改為四站，並明示「每週品牌與異動以當週公告為準」。
- 兩週內建立一份真正的營運主檔，不再用永久 `星期幾` 推算品牌班表。
- 以北投做第一個數據化試點：每場記錄預定／實到、營業時間、銷售區間、缺貨、客流抽樣、異常與顧客問題。
- 新北先做桌面篩選與場站洽談，不先承諾固定站；通過現勘、合約、供給與 4–8 週小規模試營運後再轉常態。
- 把現有 Astro 官網保留為快速、低成本的公開內容層；另外建立受權限控管的營運資料層，兩者不要混成一個巨型系統。

## 2. 研究方法與可信度

### 2.1 來源層級

| 層級 | 來源 | 本研究用途 | 注意事項 |
|---|---|---|---|
| A | 點點綠 Facebook 當週公告、主辦方現行頁面 | 現行站點、時段、活動、臨時異動 | Facebook 可見性與搜尋索引不穩定；頁面簡介不是日期化公告 |
| A | 臺北捷運、臺北市政府、農糧署 | 合作成立、政策定位、政府合作、法規與公開數據 | 新聞稿常是單次活動，不等於長期營業規則 |
| B | 點點綠招募頁、主辦／協辦媒體 | 資格、席位、經營背景、品牌與營運觀察 | 帶有招募或推廣目的，數字需內部資料再驗證 |
| B | 攤友官方頁面 | 品牌品項、個別出攤 | 只代表個別品牌，不代表整體市場規則 |
| C | 搜尋摘要、轉載媒體 | 找線索、確認曝光範圍 | 不單獨作為重大決策依據 |

### 2.2 查核原則

- 日期越新、越接近原始發布者者優先。
- 將「長期規則」「特定週班表」「單次活動」分開，不互相推論。
- 找到矛盾時保留兩邊來源與日期，不自行選一個好看的版本。
- 新聞或社群列出的營業額、攤數與席位視為觀察值，不視為保證。
- 新站選址不只看進出站人次；還需出口動線、可用空間、裝卸、雨備、站務、安全、客群、供給與單位經濟。

## 4. 目前營運樣貌

### 4.1 兩種不同產品

| 面向 | 週間 Green to Go | 雙連假日市集 |
|---|---|---|
| 核心任務 | 通勤途中快速補貨 | 逛市集、體驗、品牌故事與主題活動 |
| 時間 | 週一至週五 14:00–19:00 | 週六 14:00–20:00、週日 14:00–19:00 |
| 站點 | 雙連、士林、石牌、北投 | 雙連站 1 號出口廣場為主；粉專簡介另提 1／2 號出口，須向主辦方確認實際分工 |
| 規模 | 公開文章稱每站約 3–5 攤 | 品牌選擇與活動較多 |
| 主力 | 生鮮蔬果、蛋奶、農產加工、日常食品 | 農產、加工、永續手作、試吃、課程、主題活動 |
| 顧客意圖 | 便利、立即、帶了就走 | 發現、交流、親子、社群與休閒 |
| 成功指標 | 出席率、銷售／攤時、售罄、回購、通勤轉換 | 人流、停留、活動參與、客單、品牌曝光、再訪 |

### 5.2 媒體能見度判讀

目前搜尋結果高度集中在四類：主辦／協辦網站、政府新聞稿、農糧署案例、活動轉載。真正由獨立新聞編輯部所做、包含顧客訪談、攤友損益、營運挑戰或第三方成效評估的深度報導很少。

這不代表市集缺乏價值，而是目前的「可被搜尋故事」主要由合作網絡自己發布。若要拓展新北、企業合作或政府專案，建議準備可被第三方引用的媒體包：

- 一頁式事實表：成立、站點、場次、合作農友、累計服務、聯絡窗口。
- 3–5 個可驗證案例：農友新增通路、消費者回購、減塑量、地方／公益合作。
- 可公開的季度數據，不只活動照片與貼文觸及。
- 品牌照、攤車照、站點照之授權與圖說。
- 對外說法明確區分「有機驗證」「友善耕作」「農藥減量承諾」「一般永續手作」。

## 6. 此模式需要的管理能力

### 6.1 管理範圍

| 管理域 | 每週／每月工作 | 沒做好會發生的問題 |
|---|---|---|
| 治理與合作 | 與捷運、農糧署、政府、場地方確認權責、使用範圍與報告 | 場地誤用、承諾不一致、合作風險 |
| 攤友招募與審核 | 申請、資格分類、文件審核、規約、照片授權、上架／停權 | 不符合展售資格、宣稱風險、名單失真 |
| 日期型排班 | 站點容量、品類平衡、候補、確認、請假、換站 | 重複排班、空攤、顧客撲空 |
| 現場營運 | 進場、裝卸、動線、消防／用電、整潔、收攤、站務交接 | 公共安全、捷運營運與品牌風險 |
| 食品與農產合規 | 驗證有效期、標示、溫控、食品業者登錄、產品責任與追溯 | 稽查、食安、錯誤宣稱 |
| 顧客服務 | 價格／品項、退款爭議、客訴、失物、緊急事件 | 口碑下降、責任無人承接 |
| 天候與事件 | 颱風、豪雨、高溫、停班、捷運事件、臨時停駛 | 多站訊息不同步、農友損失 |
| 內容與媒體 | 週表、臨時公告、品牌故事、合作露出、回覆訊息 | Facebook 與官網互相矛盾 |
| 財務與單位經濟 | 攤租、優惠、成本、付款、專案收入、各站損益 | 擴點越多虧損越大 |
| 數據與改善 | 出席、銷售區間、售罄、客流、觸及、活動轉換 | 只能靠感覺選站與招商 |

### 6.2 建議角色

小團隊可以一人兼多職，但責任不可消失。

| 角色 | 對結果負責 | 可兼任／外包 |
|---|---|---|
| 計畫負責人 | 定位、預算、合作、擴點決策、重大事件 | 不宜完全外包 |
| 多站營運主管 | 班表、站務、現場標準、事件指揮 | 可與計畫 PM 兼任至一定規模 |
| 攤友與合規管理 | 申請、文件、規約、教育、品類與關係維護 | 可由行政支援，但審核責任要明確 |
| 站點值勤／巡場 | 簽到、動線、照片、事件、收攤 | 可由兼職、志工或輪值；需訓練與回報工具 |
| 內容與社群 | 週表、貼文、客服、媒體素材 | 可外包製作，不宜外包事實核准 |
| 財務與專案管理 | 收費、對帳、合作專案、站點損益 | 可由協會行政兼任 |
| 產品／資料管理 | 主檔、權限、報表、自動化、網站 | 初期可兼任或專案制，須有內部資料 owner |

### 6.3 人力負荷估算方式

不要用「站點數」直接猜人數，應用場次工時估算：

```text
每週現場工時 = 實際開站場次 ×（營業時數 + 進撤場時數）× 每場必要值勤人數
```

例如，若四站在五個週間都開、每場 5 小時、進撤場合計 1.5 小時、每站至少 1 位值勤，僅現場就會是 `4 × 5 × 6.5 = 130 人時／週`，還沒包含招商、排班、內容、客服與財務。這是容量示例，不代表目前實際開站日。它說明四站不能只靠一位中央小編／管理者臨時救火；需明確採取「站點值勤＋中央調度」或「攤友自治＋巡場稽核」模式。

### 6.4 建議的營運節奏

| 時點 | 流程 | 產出 |
|---|---|---|
| 每月前 3–4 週 | 蒐集可出席日期、品項與證明異動 | 攤友 availability |
| T-14 | 依站點容量、品類與績效排草案 | 草案班表、候補 |
| T-7 | 攤友確認、缺口補位、站務確認 | `confirmed` 場次 |
| T-3～T-2 | 一次核准，發布官網／社群週表 | 對外版本與版本號 |
| T-1 | 天候、交通、品項、設備再確認 | go／hold／cancel |
| 當日進場 | QR 或簡易表單簽到、攤位照片 | 實到與開場時間 |
| 營業中 | 客流抽樣、缺貨、客訴、事件 | 現場紀錄 |
| 收攤 | 簽退、清潔、銷售區間、明日異動 | 場次結案 |
| 每週 | 15–30 分鐘例會 | 問題清單、下週調整 |
| 每月 | 站點與攤友回顧 | 續留、輔導、擴縮決策 |

### 6.5 合規與安全清單

公開招募頁已表明捷運用地重視公共安全，申請者需遵守展售資格、營業規範並簽署規約／同意書；有機、友善、綠保、動福、永續漁業、CAS、TAP 與生產追溯資料可作審核依據。建議營運系統至少追蹤：

- 主體／攤位負責人、聯絡方式、緊急聯絡人。
- 展售品項與品類；生鮮、加工、即食、酒類、肉蛋奶分開管理。
- 有機／友善／綠保／TAP 等證明類別、檔案、發證者、有效期與查核日期。
- 無證明產品的可使用宣稱；若採農藥減量承諾，保留簽署版本與日期。
- 食品業者登錄、產品責任保險或場地方要求的其他文件。
- 加工食品標示、過敏原、保存方式、有效日期、冷鏈需求。
- 用電、火源、設備、攤車尺寸、動線、裝卸方式。
- 攤友規約、影像／文案授權、個資告知與資料保留期限。
- 客訴、食安、受傷、財損、設備、站務等事件分類與升級機制。

有機農產品的容器、包裝與散裝展示均受標示規範；市集也可能被主管機關不定期查驗。網站與社群不應因行銷方便，將「友善」「無毒」「有機」「自然農法」混成同一種法律或驗證狀態。

## 7. 北投站：從上線轉為可衡量試點

### 7.1 現在的正確問題

北投已於 2026-07-01 初登場，因此應問：

- 哪幾天實際開站？每日容量與實到率多少？
- 哪些品類在北投比其他站更適配？
- 通勤客、居民、轉乘新北投／觀光客比例如何？
- 站外動線、雨熱、裝卸與收攤有何摩擦？
- 新站是否增加總銷售與農友收益，或只是把同一批供給從其他站移過去？
- 管理工時與場地成本是否被單攤產出覆蓋？

### 7.2 建議 6–12 週基線

| 類別 | 每場最低紀錄 |
|---|---|
| 排班 | 預定攤數、實到攤數、遲到／早退、臨時請假 |
| 銷售 | 每攤銷售區間、交易筆數或簡化估值、售罄品項 |
| 顧客 | 三個 15 分鐘客流抽樣、停留／購買抽樣、新舊客快問 |
| 品類 | 生鮮、蛋奶、加工、烘焙、手作之攤數與產出 |
| 現場 | 進撤場時間、天候、動線、站務問題、垃圾與設備 |
| 傳播 | 貼文觸及、連結點擊、詢問、地圖／QR 掃描 |
| 成本 | 場地、交通、值勤、宣傳、物資與合作補助 |

### 7.3 90 天決策門檻

不宜只設定單一營收門檻。建議北投同時符合以下四類才由「pilot」轉「core」：

1. **可靠**：已確認場次實到率、準時率、公告正確率達團隊設定標準。
2. **有利攤友**：多數參與攤友願意續留，且每攤時產出不持續低於可接受底線。
3. **可管理**：每場管理工時、事件與站務摩擦在可承受範圍。
4. **有新增性**：吸引新客、創造新增銷售或合作，而不是明顯侵蝕相鄰站點。

門檻數字應用前 4–6 週基線與三個既有站比較後設定，不宜現在憑空指定。

## 8. 新北擴站策略

### 8.1 擴站前置 Gate

| Gate | 必須完成 | 不通過時 |
|---|---|---|
| G0 內部可複製 | 四站角色、班表、文件、事件、財務有共同 SOP | 暫停新增常態站 |
| G1 需求證據 | 官方運量、出口分時客流、商圈／住宅、訪談、預購或快閃測試 | 回到桌面研究 |
| G2 場地可行 | 業主、政府、站務、空間、裝卸、用電、雨備、安全均可接受 | 換出口或換站 |
| G3 供給可行 | 至少有一組穩定農友池與候補，不拉空既有站 | 做主題快閃而非常態 |
| G4 單位經濟 | 攤友產出、場地與管理成本、合作資源可持續 | 調整時段／費用／規模 |
| G5 試營運 | 4–8 週小規模、每週檢討、事件可控 | 關閉或延長試驗 |
| G6 常態化 | KPI、合約、排班、站務、內容均穩定 | 才納入常態站點 |

### 8.2 站點評分卡

| 維度 | 權重 | 查核項目 |
|---|---:|---|
| 需求與客群適配 | 25 | 平日 14–19 分時客流、住宅／辦公、家庭、健康／生鮮需求、回家動線 |
| 場地與現場可行 | 20 | 出口可見性、通行淨寬、攤位容量、遮雨、熱環境、用電、垃圾、裝卸 |
| 合作與許可 | 15 | 場站業主、地方政府、站務、商圈、里辦、合約期與費用 |
| 供給與單位經濟 | 15 | 農友距離、品類、候補、物流、攤租、預估每攤時產出 |
| 安全與可及性 | 10 | 無障礙、消防、行人衝突、食品溫控、夜間照明 |
| 地方網絡與新增性 | 10 | 新北青農、社區、學校、企業、既有市場互補、對舊站侵蝕 |
| 數據與傳播可行 | 5 | 可計數、QR／地圖入口、站方宣傳、資料回收 |

建議總分 75 以上才進 4–8 週試營運；60–74 補資料或做一次性快閃；低於 60 不進場。任何公共安全、合約或供給 Gate 不通過，即使總分高也不應開站。

### 8.3 新北的建議拓展方式

1. **先拓合作，再拓固定站。** 已有新北青農主題活動訊號，可先做「產地週／區域週」測試供給、故事與顧客反應。
2. **先做一個站群，不做全市散點。** 優先選與現有紅線物流或農友路線能共享的區域；集中巡場、物資與候補，降低管理半徑。
3. **分清場站業主。** 新北境內可能由臺北捷運或新北捷運營運，合約、站務與資料窗口不一定相同，不能沿用一份臺北站點規約。
4. **用官方分時運量而非全年總人次。** 目標是週間 14–19 點的「回家前可停留」客流，不是單純最大站。
5. **先收意向再選站。** 可用一頁式候選站調查、QR 候補名單或預購取貨測試，確認顧客願意在哪站停下來。
6. **避免與既有市場正面衝突。** 對傳統市場、黃昏市場與地方農會，以補時段、補品類、聯合活動或導客為定位。

目前不建議僅憑桌面資料指定「新北第一站」。官方各站分時運量、出口現勘、場地方意願、農友物流與四站實際數據尚未合併，現在點名會造成錯誤確定感。

## 10. 軟體與資訊科技的角色

### 10.1 正確的系統邊界

目前官網規格把網站定位為介紹、品牌牆與招募導流，MVP 不做電商、會員或後台，這仍是合理決策。新的營運需求應拆成兩層：

```text
營運主檔／排班／合規／事件（受權限控管）
                    ↓ 核准發布
     官網　Facebook　週表圖卡　合作報表
                    ↓ 回收
      點擊／詢問　簽到　銷售區間　事件
```

- **公開層**：現有 Astro 官網，快、便宜、SEO 友善，顯示已核准資料。
- **營運層**：有登入、權限、版本、文件與日期型排班的資料系統。
- **整合層**：把核准場次產生官網資料、圖卡草稿、行事曆與通知。
- **分析層**：站點、場次、攤友、活動與合作成效。

不要以爬 Facebook 作為正式資料流程。最穩定的方法是先在內部系統核准，再由管理者發布到 Facebook；若使用 Meta API，須由粉專管理者依官方權限與審查流程串接，並保留人工發布備援。

### 10.2 第一版資料模型

| Entity | 核心欄位 | 用途 |
|---|---|---|
| `Organization` | 主辦、場站、政府、合作單位、聯絡窗口 | 多方合作與權責 |
| `Station` | 系統、站號、出口、容量、時段、場地方、狀態 | 擴站與場地管理 |
| `MarketSession` | 日期、開始／結束、站點、類型、狀態、天候、來源版本 | 日期型場次，不用永久星期推算 |
| `Vendor` | 法定／品牌名、類別、聯絡、狀態、故事、授權 | 攤友主檔 |
| `Credential` | 類型、號碼、發證者、有效期、檔案、查核狀態 | 合規與到期提醒 |
| `Product` | 品項、品類、宣稱、標示、保存、產季、來源 | 品類平衡與內容 |
| `Assignment` | 場次、攤友、攤位、確認、簽到、簽退、候補 | 排班與出席 |
| `Incident` | 場次、類型、嚴重度、處理者、證據、結案 | 颱風、食安、客訴、站務 |
| `Campaign` | 合作方、目標、場次、素材、預算、指標 | ESG 與主題專案 |
| `Metric` | 客流抽樣、交易／銷售區間、售罄、掃碼、減塑 | 評估與擴站 |
| `ContentItem` | 渠道、文案、圖、事實版本、核准者、發布網址 | 多管道一致性 |
| `FormLink` | 用途、網址、有效期、owner、狀態 | 解決表單連結散落 |

### 10.3 科技導入順序

#### Phase 0：0–4 週，資料可信

- 指定每個資料域 owner。
- 清理四站、攤友、表單、聯絡、資格與現行狀態。
- 建立日期型班表與 `draft / confirmed / cancelled / completed` 狀態。
- 每筆公開資料保留 `sourceUrl / updatedAt / approvedBy`。
- 官網只顯示已確認場次；沒有確認資料時直接說明，不自動推算。
- 建立颱風／停班／站務事件的一鍵公告模板。

#### Phase 1：1–3 個月，四站營運 MVP

- 管理者登入、角色權限、修改紀錄。
- 攤友 availability、排班、候補、確認與提醒。
- 證明文件到期提醒與查核佇列。
- 現場手機簽到／簽退、照片、事件回報。
- 自動產生週表網頁、社群文案草稿與圖卡資料。
- 北投基線 dashboard；按站點比較可靠度與單位經濟。

#### Phase 2：3–6 個月，顧客與新業務

- 品牌／站點 QR、收藏與到站提醒。
- 小規模預購／保留，先不做複雜庫存；清楚定義付款與取消。
- 企業／社區團購與站點取貨。
- Campaign 成效、合作報告、媒體素材庫。
- 顧客回饋與新北候選站意向收集。

#### Phase 3：6–12 個月，多區域複製

- 臺北／新北不同場站業主、規約、收費與權限。
- 標準化站點啟動包、試營運與關站流程。
- API／資料匯出、開放資料與官方運量整合。
- 經核准內容提供 MCP／AI 助理查詢；禁止 AI 把草稿當事實。
- 站點、攤友與合作組合的預測與情境模擬。

### 10.4 AI 可以做與不能做的事

**適合：**

- 由已核准班表生成 Facebook、官網、LINE 與圖卡文案草稿。
- 品牌資料去重、缺欄位提示、證明到期摘要。
- 把事件回報分類、產生週報與待辦。
- 依實際數據產生站點比較與擴站情境。
- 將農友訪談轉成品牌故事草稿，經本人同意後發布。

**不應自動決定：**

- 攤友是否符合有機／友善資格。
- 食安、醫療、法律與公共安全事件處置。
- 因付費、關係或 AI 分數自動停權／排除攤友。
- 從舊貼文推測當週仍出攤。
- 未經人工核准直接對外發布營業、休市或危機訊息。

## 11. KPI 與儀表板

### 11.1 北極星與護欄

建議北極星不是單純人流，而是：

> **每個可持續營運場次所創造的攤友有效銷售與穩定回購。**

同時用公共安全、合規、攤友公平與資料正確作護欄。

### 11.2 KPI 分層

| 層級 | 建議指標 |
|---|---|
| 可靠度 | 已確認場次實到率、準時率、取消提前量、公告正確率、事件結案時間 |
| 攤友價值 | 銷售／攤時、售罄率、續留意願、新客名單、跨站增量、付款準時 |
| 顧客價值 | 交易／客流抽樣、回購、客單區間、找得到率、QR 回訪、抱怨率 |
| 站點經濟 | 站點貢獻、管理工時／場、場地與宣傳成本、對鄰站侵蝕 |
| 品類健康 | 生鮮占比、品類缺口、同質攤位、季節供應、價格帶 |
| 合規安全 | 文件完整率、到期未處理、稽查問題、食安／受傷／站務事件 |
| 傳播 | 週表準時率、觸及、點擊、詢問轉換、媒體引用、資料更新時間 |
| 永續影響 | 農友直接銷售、在地／友善產品、減塑行動、合作參與、公益成果 |

不要要求攤友一開始回報精確到元的營收。可先用自願、保密的銷售區間與交易數，建立信任；對外只發布彙總且達到最小樣本數的資料。

## 12. 90 天行動計畫

### 第 1–14 天：把事實對齊

- 指定營運、攤友、內容、財務、科技 owner。
- 確認四站實際開站日、容量、出口、站務與值勤方式。
- 確認兩個週末出口的實際用途。
- 統一加入表單、場地表單與聯絡窗口。
- 盤點有效攤友與資格文件；歷史、暫停、候補分開。
- 官網更新北投並修正仍寫「三站／即將試營運」的內容。

### 第 15–30 天：建立營運主檔

- 建立上述核心 entity 的簡化版本。
- 以日期＋站點＋攤友管理 4 週班表。
- 建立批准、發布、取消與版本流程。
- 做手機簽到、異常、銷售區間與客流抽樣表。
- 設定資料權限、備份、匯出與個資保留規則。

### 第 31–60 天：量測北投與自動發布

- 每場收集北投基線；同時抽樣一個成熟站作對照。
- 週表從同一資料源產出官網、社群草稿與內部名單。
- 上線資格到期提醒與未確認排班警示。
- 每週檢討出席、銷售區間、事件、品類與顧客問題。

### 第 61–90 天：決定北投常態化與新北下一步

- 用 6–8 週資料判斷北投為 core、extended pilot、seasonal 或 pause。
- 用官方分時運量、現勘與合作意願做新北 5–8 站長名單。
- 評分後選 2–3 站進一步洽談，不直接宣布。
- 先做新北青農／地方物產主題活動或預購取貨測試。
- 完成新北試營運包：合約清單、SOP、資料、KPI、關站條件。

## 13. 需要主辦方回答的問題

### 現況

1. 四站各自固定開哪些日？是否每個週間都同時開四站？
2. 北投目前是短期試營運、季節性或預定常態？評估日期與條件為何？
3. 雙連 1 號與 2 號出口的假日／平日／免費場地用途如何分工？
4. 目前有效的加入表單、舊夥伴表單與場地表單各是哪一個？
5. 對外應使用協會聯絡方式或 Facebook 所列窗口？各自服務什麼問題？

### 營運與財務

6. 每站容量、場地費、攤租、優惠、值勤、裝卸與設備規則為何？
7. 一個月實際產生多少可售席位、排滿率、實到率與候補？
8. 攤友最小可接受銷售、交通與工時是多少？哪些站點已達可持續？
9. 臨時請假、休市、客訴、食安與公共安全事件由誰決策與通知？
10. 是否可蒐集銷售區間、交易數與顧客回饋？資料可公開到何種程度？

### 科技與資料

11. 目前班表、攤友、文件、收費與貼文各放在哪裡？誰是 owner？
12. 哪些人需要看／改哪些資料？攤友是否需要自助回報 availability？
13. 是否有粉專管理權限與 Meta API 串接意願？沒有 API 時誰負責人工發布？
14. 品牌文案、照片、社群連結與對外成效資料是否已有授權？
15. 電商／預購的真正問題是顧客需求、缺貨、天候風險，還是提高客單？

## 14. 主要風險與對策

| 風險 | 早期訊號 | 對策 |
|---|---|---|
| 社群成為唯一資料庫 | 官網、表單、貼文不同步 | 內部主檔核准後再多管道發布 |
| 擴點快於供給 | 舊站空攤、同一批攤友疲勞 | 供給 Gate、候補池、分群擴站 |
| 合規宣稱混用 | 所有攤友都寫成有機／無毒 | Credential 與可用宣稱分離、人工查核 |
| 管理者過載 | 每週靠私訊補洞、休假即停擺 | RACI、固定節奏、提醒與站點值勤 |
| 營收好看但不賺 | 攤友銷售增加、中央成本更快增加 | 站點貢獻與管理工時一起看 |
| 新站侵蝕舊站 | 總銷售不增、攤友只換位置 | 顧客／銷售新增性與重疊追蹤 |
| 危機訊息延遲 | 颱風當天才逐一通知 | 事件狀態、一鍵模板、多管道回執 |
| 過早做完整電商 | 訂單少、維護與退款成本高 | 先做預留／團購／取貨小實驗 |
| AI 產生錯誤事實 | 舊班表被寫成當週公告 | 僅讀核准資料、人工發布、保留來源 |
| 數據傷害信任 | 攤友擔心營收被比較或外洩 | 最小蒐集、用途告知、區間化、彙總公開 |

## 16. 文件維護規則

- 現行營業與站點資訊至少每月查一次；颱風、試營運與擴站期間每週查。

- 研究結論若引用內部營收、人力與合約，應另建內部版本，不把個資與商業機密提交公開 repo。
- 新北候選站只有在評分卡、現勘與合作意向完成後，才從「研究」升為「計畫」。
